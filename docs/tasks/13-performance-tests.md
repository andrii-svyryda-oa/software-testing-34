# Task 13 — Performance tests (k6)

## Goal

Two k6 scripts targeting the most performance-sensitive endpoints:

- **Load test** on the point-balance lookup (`GET /api/customers/{id}`).
- **Stress test** on concurrent redemption (`POST /api/customers/{id}/redeem`) of a deliberately scarce reward.

Tests run against a live API instance with the database pre-seeded by `LoyaltySeeder` (≥ 10 000 records) so query patterns reflect realistic data volume.

```

perf/redeem-stress.js
```

## `balance-load.js` (point-balance lookup)

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Trend } from 'k6/metrics';

const latency = new Trend('balance_lookup_ms');

export const options = {
    stages: [
        { duration: '30s', target: 50 },     // ramp-up
        { duration: '2m',  target: 200 },    // sustained 200 VU
        { duration: '30s', target: 0 },      // ramp-down
    ],
    thresholds: {
        'http_req_duration': ['p(95)<200'],   // p95 < 200ms
        'http_req_failed':   ['rate<0.01'],   // <1% errors
        'balance_lookup_ms': ['p(99)<500'],
    },
};

const baseUrl = __ENV.API_URL || 'http://localhost:5000';
const customerIds = JSON.parse(open(`${__ENV.IDS_FILE || './customer-ids.json'}`));

export default function () {
    const id = customerIds[Math.floor(Math.random() * customerIds.length)];
    const t0 = Date.now();
    const res = http.get(`${baseUrl}/api/customers/${id}`);
    latency.add(Date.now() - t0);

    check(res, {
        'status 200': (r) => r.status === 200,
        'has total_points': (r) => r.json('totalPoints') !== undefined,
    });

    sleep(0.1);
}
```

`customer-ids.json` is generated by the seeder (a list of all seeded customer GUIDs). Pass via `IDS_FILE` env var.

## `redeem-stress.js` (concurrent redemption stress)

```javascript
import http from 'k6/http';
import { check } from 'k6';
import { Counter } from 'k6/metrics';

const successes = new Counter('redeem_success');
const stockExhausted = new Counter('redeem_stock_exhausted');
const insufficient = new Counter('redeem_insufficient');

export const options = {
    scenarios: {
        burst: {
            executor: 'shared-iterations',
            vus: 100,
            iterations: 1000,
            maxDuration: '1m',
        },
    },
    thresholds: {
        // No 5xx allowed: failures must be a clean 422 / 404.
        'http_req_failed': ['rate<0.005'],
    },
};

const baseUrl = __ENV.API_URL || 'http://localhost:5000';
const richCustomers = JSON.parse(open('./rich-customer-ids.json')); // pre-seeded customers with >> 100k points
const scarceRewardId = __ENV.REWARD_ID; // a reward with StockQuantity = 50

export default function () {
    const customerId = richCustomers[Math.floor(Math.random() * richCustomers.length)];
    const res = http.post(
        `${baseUrl}/api/customers/${customerId}/redeem`,
        JSON.stringify({ rewardId: scarceRewardId }),
        { headers: { 'content-type': 'application/json' } }
    );

    if (res.status === 200) successes.add(1);
    else if (res.status === 422) {
        const body = (res.body || '').toString();
        if (body.includes('out of stock')) stockExhausted.add(1);
        else if (body.includes('only')) insufficient.add(1);
    }

    check(res, {
        'status is 200/422/404': (r) => [200, 422, 404].includes(r.status),
        'no server error': (r) => r.status < 500,
    });
}
```

Post-run **manual** assertion (k6 doesn't enforce this; the perf README documents it):

> `redeem_success` should equal the initial `StockQuantity` of the scarce reward (e.g. 50). Anything higher means we have an over-redemption bug.

## `seed-bootstrap.cs`

A tiny `dotnet run` console (or just an extra `--seed` CLI flag in the `Api` project) that:

1. Connects to the configured Postgres.
2. Calls `LoyaltySeeder.SeedAsync(...)` with `targetTotal = 10_000, seed = 42`.
3. Writes `customer-ids.json` (all seeded customer GUIDs) and `rich-customer-ids.json` (only customers with `TotalPoints > 100_000`).
4. Inserts a "scarce" reward with `StockQuantity = 50` and writes its GUID to stdout (consumed by `REWARD_ID` env var in the k6 run).

## `docker-compose.perf.yml`

```yaml
services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: loyalty
    ports: ['5432:5432']

  api:
    build:
      context: ..
      dockerfile: Api/Dockerfile      # add a minimal Dockerfile in the Api project

  k6:
    image: grafana/k6:latest

    environment:
      API_URL: 'http://api:5000'
    depends_on: [api]
```

## `perf/README.md` content

Document:

1. **Prerequisites**: Docker, `k6` CLI (or Docker), .NET SDK 8.
2. **Local run**:
   ```bash
   docker compose -f perf/docker-compose.perf.yml up -d postgres api
   dotnet run --project Api -- --seed > perf/scarce-reward-id.txt
   k6 run -e API_URL=http://localhost:5000 perf/balance-load.js
   k6 run -e API_URL=http://localhost:5000 \
          -e REWARD_ID=$(cat perf/scarce-reward-id.txt) \
          perf/redeem-stress.js
   ```
3. **Pass criteria**:
   - `balance-load.js`: `http_req_duration p(95) < 200ms`, error rate < 1 %.
   - `redeem-stress.js`: `redeem_success == 50` (scarce reward stock), no 5xx, all failures are clean 422.

## Acceptance criteria

- Both `.js` scripts run via the standalone `k6` binary (`k6 run perf/<script>.js`) without errors.
- The seeder produces a file with ≥ 500 customer IDs and ≥ 1 scarce reward ID.
- `redeem-stress.js` results show **no over-redemption** when run against the API on a developer laptop.
